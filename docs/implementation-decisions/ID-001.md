# ID-001 - Reset Spring Context After Integration Tests

## Status

Accepted

## Problem

When running the integration tests individually, all of them passed successfully. However, executing the entire test suite with `mvn test` caused some tests to fail due to state being shared between test classes.

Spring Boot caches the application context to improve test performance. As a result, modifications made during one integration test could affect subsequent tests that reused the same context, leading to non-deterministic failures.

## Decision

Annotate every integration test class with:

```java
@DirtiesContext(classMode = DirtiesContext.ClassMode.AFTER_CLASS)
```

This instructs Spring to mark the application context as dirty after the test class has finished, forcing the framework to discard it and create a fresh context for subsequent tests.

## Alternatives Considered

### 1. Cleaning the database manually after each test

Although this would remove persisted data, it would not reset other beans or application state that may have been modified during the tests.

**Rejected** because it does not guarantee complete test isolation.

### 2. Using `@Transactional` with automatic rollback

This approach works well for many repository tests, but it is not suitable for all integration tests, especially those involving multiple transactions or full HTTP request lifecycles.

**Rejected** because it does not cover every integration testing scenario.

### 3. Resetting individual beans manually

Specific beans could be reset after each test.

**Rejected** because it increases maintenance effort and becomes difficult as the application grows.

## Consequences

### Advantages

- Ensures complete isolation between integration test classes.
- Prevents hidden dependencies between tests.
- Guarantees consistent behavior when executing the full test suite with `mvn test`.
- Makes the CI pipeline more reliable.

### Disadvantages

- Spring must recreate the application context after each integration test class.
- Test execution time increases slightly due to the additional context initialization.

## Rationale

Although recreating the Spring context introduces a small performance overhead, the improvement in reliability, reproducibility, and test independence outweighs the additional execution time. Since integration tests prioritize correctness over speed, ensuring a clean application context after each test class was considered the most robust solution.
