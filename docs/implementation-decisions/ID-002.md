# ID-002 - Make SSL Configurable for CI/CD Workflows

## Status

Accepted

## Problem

The production backend is configured to use HTTPS with a self-signed SSL certificate. Consequently, integration tests executed against a deployed backend must communicate over HTTPS instead of HTTP.

However, the self-signed certificate intended for production deployment is protected with a strong password and should not be exposed or reused in CI/CD workflows. This created the need for a different strategy to support automated testing while preserving the security of the production configuration.

## Decision

Make SSL support configurable through an environment variable.

When SSL is enabled, the application starts using HTTPS and the configured certificate. When SSL is disabled, the backend runs over HTTP.

This allows GitHub Actions workflows to execute the integration tests without requiring any certificate, while production deployments simply enable SSL through the corresponding environment configuration.

## Alternatives Considered

### 1. Create a second self-signed certificate for CI/CD

Generate an additional self-signed certificate protected with a simple password and use it exclusively for GitHub Actions, while keeping the production certificate secured with a strong password.

**Rejected** because it requires maintaining two different certificates and introduces additional configuration and maintenance overhead. It also increases the risk of inconsistencies between testing and production environments.

### 2. Make SSL optional through an environment variable

Configure the application so that SSL can be enabled or disabled depending on the execution environment.

GitHub Actions execute the backend with SSL disabled, while production deployments enable SSL using the production configuration.

**Accepted** because it avoids maintaining multiple certificates while keeping the production certificate isolated from the CI/CD pipeline.

## Consequences

### Advantages

- The production certificate is never exposed to the CI/CD environment.
- No additional certificates need to be generated or maintained.
- Integration tests can run using HTTP without additional SSL configuration.
- Production deployments continue to use HTTPS by simply enabling SSL through environment variables.
- The deployment configuration remains simple and easy to maintain.

### Disadvantages

- The CI/CD environment does not execute integration tests over HTTPS.
- An additional environment property must be managed to enable or disable SSL depending on the execution environment.

## Rationale

The objective was to avoid exposing the production SSL certificate while keeping the testing infrastructure simple and maintainable. Making SSL configurable through an environment variable achieves both goals: production deployments remain secure using HTTPS, whereas automated workflows can execute without managing certificates or sensitive credentials. This solution minimizes maintenance effort while preserving a clear separation between testing and production environments.
