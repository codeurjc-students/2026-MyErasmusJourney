# DD-001 - Service Layer Decoupled Through Factory Functions

## Problem

A decision had to be made regarding how React components should access the REST API.

## Alternatives Considered

### Option A

Import HTTP request functions directly into each component.

### Option B

Export a global service instance.

### Option C (Chosen)

Create a factory function (`createExperienceService`, `createUniversityService`, etc.) that receives an `ApiClient` and returns an object containing all service methods.

## Decision

Each domain exposes a `createXService(apiClient)` function that creates a service instance containing all related operations.

## Rationale

- Decouple business logic from the HTTP client implementation.
- Simplify unit testing through dependency injection and mocks.
- Allow the HTTP client implementation (Fetch, Axios, etc.) to be replaced without affecting the service interface.
- Follow the Dependency Inversion Principle.

## Consequences

### Advantages

- Services are easy to test.
- Low coupling.
- High reusability.
- The HTTP client implementation can be replaced without modifying React components.

### Drawbacks

- Introduces a small abstraction layer.
