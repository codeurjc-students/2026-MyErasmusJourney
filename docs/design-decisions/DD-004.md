# DD-004 - Service Instances vs Independent Functions

## Problem

A decision had to be made regarding how REST API operations should be organized and how shared configuration—particularly the API base URL—should be managed.

## Alternatives Considered

### Option A

Export independent functions.

```ts
getExperiences();
getExperience(id);
createExperience(data);
```

### Option B (Chosen)

Group related operations inside a service instance.

```ts
const experienceService = createExperienceService(apiClient);

experienceService.getAll();
experienceService.getById(id);
experienceService.create(data);
```

## Decision

Each domain exposes a factory function that receives an `ApiClient` and returns a service instance.

The application (`web`) creates these instances using the API base URL obtained from environment variables.

## Rationale

The main motivation was to maintain a clear separation between application configuration and API implementation.

Environment variables belong to the application (`web`), not to the shared library (`shared`).

As a result:

- `shared` does not depend on Vite.
- `shared` does not use `import.meta.env`.
- Configuration changes do not require modifications to the service implementation.
- The shared library can be reused by other applications or in testing environments.

Additionally, grouping related operations into a single service improves code organization and ensures that React components depend only on the service's public interface rather than its implementation details.

## Consequences

### Advantages

- Clear separation between configuration and implementation.
- `shared` remains framework-agnostic.
- Improved reusability.
- The `ApiClient` can easily be replaced during testing.
- Related operations are grouped together.
- Components depend only on the service contract, not on its implementation.

### Drawbacks

- A service instance must be created before it can be used.
- Introduces a small abstraction layer compared to importing standalone functions.
